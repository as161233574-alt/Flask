## 一、数据模型（app/models/user.py）

导入路径必须用两个点：from ..extensions import db
（因为 user.py 在 models/ 子包内，需向上两层才能到 app/ 包的 extensions）

### User 模型（表名 users）
字段：
  id            INTEGER  PRIMARY KEY AUTOINCREMENT
  username      VARCHAR(80)   UNIQUE NOT NULL
  email         VARCHAR(120)  UNIQUE NOT NULL
  password_hash VARCHAR(256)  NOT NULL
  role          VARCHAR(20)   NOT NULL DEFAULT 'user'   ← 只允许 'user' | 'admin'
  created_at    DATETIME      DEFAULT = lambda: datetime.now(timezone.utc)
  updated_at    DATETIME      DEFAULT + onupdate = lambda: datetime.now(timezone.utc)

方法：
  set_password(plain)   → generate_password_hash(plain)
  check_password(plain) → check_password_hash(self.password_hash, plain)
  to_dict()             → 返回 {id, username, email, role, created_at（.isoformat()）}
                          不包含 password_hash

### TokenBlocklist 模型（表名 token_blocklist）
字段：
  id         INTEGER  PRIMARY KEY AUTOINCREMENT
  jti        VARCHAR(36)  UNIQUE NOT NULL
  created_at DATETIME     DEFAULT = lambda: datetime.now(timezone.utc)

## 五、API 端点

### 统一响应格式
成功：{"success": true,  "data": {...}}
失败：{"success": false, "error": "描述"}
状态码：200 成功 / 201 创建 / 400 参数错误 / 401 未认证 / 403 无权限 / 404 未找到 / 409 冲突

### 认证路由（auth_bp，前缀 /auth）

POST /auth/register — 无需认证
  body: {username, email, password}
  校验：三字段不能为空；password >= 6 位；username 和 email 不得重复
  成功：201 + user.to_dict()

POST /auth/login — 无需认证
  body: {email, password}
  校验：两字段不能为空；邮箱+密码组合正确
  成功：200 + {access_token, refresh_token}

POST /auth/logout — @jwt_required()
  取 get_jwt()["jti"] 写入 TokenBlocklist
  成功：200 + {message: "logged out"}

POST /auth/refresh — @jwt_required(refresh=True)
  成功：200 + {access_token}（新签发）

### 用户路由（user_bp，前缀 /users）

GET  /users/me — @jwt_required()
  返回当前用户 to_dict()

PUT  /users/me — @jwt_required()
  body: {username?, email?}（均可选，有值才更新）
  更新前检查唯一性冲突，成功返回 200 + user.to_dict()

PUT  /users/me/password — @jwt_required()
  body: {current_password, new_password}
  校验：两字段不能为空；current_password 正确；new_password >= 6 位
  成功：200 + {message: "password updated"}

GET  /users/ — @admin_required
  返回所有用户 to_dict() 列表

PUT  /users/<int:user_id>/role — @admin_required
  body: {role}，只允许 "user" | "admin"
  用户不存在返回 404，成功返回 200 + user.to_dict()

## 六、admin_required 装饰器（app/utils/decorators.py）

from functools import wraps
from flask import jsonify
from flask_jwt_extended import jwt_required, get_jwt_identity
from ..models.user import User

def admin_required(fn):
    @wraps(fn)
    @jwt_required()
    def wrapper(*args, **kwargs):
        user = User.query.get(int(get_jwt_identity()))
        if not user or user.role != "admin":
            return jsonify({"success": False, "error": "admin access required"}), 403
        return fn(*args, **kwargs)
    return wrapper

## 八、Web 前端界面（app/templates/index.html）

单页应用，纯 HTML + CSS + 原生 JS，不引用任何外部 CDN，无框架依赖。

### 功能结构

未登录状态（auth-section）：
  - 登录 Tab：邮箱 + 密码输入框，登录按钮，错误提示，Tab 切换到注册
  - 注册 Tab：用户名 + 邮箱 + 密码，注册成功自动跳转登录

已登录状态（dashboard，auth-section 隐藏）：
  - 顶部导航栏：左侧 App 名称，右侧显示当前用户名 + 管理员徽章（admin 时可见）+ 退出按钮
  - Tab 1 个人信息：卡片展示 ID/用户名/邮箱/角色/注册时间，下方表单修改用户名和邮箱
  - Tab 2 修改密码：当前密码 + 新密码 + 确认密码，前端校验一致性
  - Tab 3 用户管理（仅 admin 可见）：
    - 统计卡片（总用户数 / 管理员数 / 普通用户数）
    - 用户列表表格（ID/用户名/邮箱/角色下拉框/注册时间/状态指示点）
    - 角色下拉框 onchange 直接调用 PUT /users/<id>/role

### JS 核心逻辑

const API = '';  ← 空字符串，调用相对路径（同域 API）

token 存取：localStorage.getItem/setItem('access_token') / ('refresh_token')

async function api(method, path, body, useRefresh=false)：
  自动附加 Authorization: Bearer <token>
  返回 {ok, status, data}

初始化（立即执行）：
  如果 localStorage 有 access_token，调用 GET /users/me
  成功则直接进入 dashboard（避免刷新页面重新登录）
  失败则显示 auth-section

退出登录：
  调用 POST /auth/logout（撤销 token）
  清除 localStorage，显示 auth-section

### CSS 关键设计
使用 CSS 变量定义主题色，Flexbox/Grid 布局，响应式（移动端适配）
所有样式内联在 <style> 标签中，不引用外部 CSS 文件

## 提示词设计意图详解（共 16 条）

---

### 意图一：禁止截断 + 文件标记格式

> "不得截断，不得使用省略号或 # TODO 占位符"
> "每个文件第一行写注释：# FILE: 路径"

**问题：** 模型在输出大量代码时会静默压缩，写出 `# ... rest of code`。
**方案：** 开头硬性约束 + 文件标记让输出可被脚本或人工精准解析，直接还原文件结构。

---

### 意图二：禁止 app.py — 记录真实踩坑

> "严禁在项目根目录创建 app.py，否则会与 app/ 包冲突"

**背景：** 本项目实际遇到过这个问题。Python 包解析时，`app.py` 文件优先于 `app/` 目录，
导致 `from app import create_app` 从空文件导入，所有路由丢失，服务返回 404。
明确写进提示词，让模型避免重蹈覆辙。

---

### 意图三：完整文件列表含所有空 `__init__.py`

**背景：** `app/models/user.py` 的相对导入 `from ..extensions import db`
在实际开发中被模型写成了 `from .extensions import db`（一个点），
导致 `ModuleNotFoundError: No module named 'app.models.extensions'`。
列出完整文件树并在模型说明中强调两个点，从根源消除导入路径猜测。

---

### 意图四：字段级数据模型约束

**问题：** 模型自行决定字段名时，`created_at` 有时写成 `createdAt`，`to_dict()` 的键名不稳定，
而测试代码直接用 `r.get_json()["data"]["created_at"]` 访问，名称不一致导致 `KeyError`。
**方案：** 逐字段描述，包括 `to_dict()` 的完整返回结构，测试和实现共用同一份"合同"。

---

### 意图五：相对导入路径显式说明

> "导入路径必须用两个点：from ..extensions import db（因为 user.py 在 models/ 子包内）"

这是本项目实际踩过的 bug。提示词中用括号解释了"为什么是两个点"，
让模型理解包层级关系而不是死记语法，减少类似错误的概率。

---

### 意图六：host="0.0.0.0" 的必要性说明

> "host='0.0.0.0' 确保局域网和公网隧道可访问，不写 host 默认只监听 127.0.0.1"

**背景：** Flask 默认 `app.run()` 绑定 `127.0.0.1`，手机访问时返回"无法连接"。
本项目因为这个问题导致局域网访问失败，加上注释让模型理解原因，而非单纯复制参数。

---

### 意图七：统一响应格式消除随机变量

> `{"success": true, "data": {...}}` / `{"success": false, "error": "..."}`

36 个测试函数全部依赖这个固定结构（`r.get_json()["data"]["access_token"]` 等）。
不规范时模型随机选择响应结构，导致测试随机失败，错误信息是 `KeyError` 而非语义错误，极难排查。

---

### 意图八：TestingConfig 设计——隔离而非共享

> `scope="function"` 的 client + `create_all/drop_all` 配对

测试 A 注册的用户会影响测试 B 的"用户名重复"判断。
`function` 级 fixture 配合 in-memory SQLite 保证每个测试函数完全隔离，
消除测试顺序依赖，让 `pytest -x` 和并行运行都稳定。

---

### 意图九：admin_token 的"绕过式"创建

> "在 app_context 内直接修改 DB 将 role 改为 'admin'"

**循环依赖：** "创建管理员"需要 admin 权限，但第一个 admin 怎么来？
不能通过 API 自举，必须在测试中直接操作数据库。
提示词明确说明这个反直觉的操作，否则模型会生成调用不存在 API 的 fixture。

---

### 意图十：按分支枚举测试——强制 100% 覆盖

**错误做法：** "测试登录" → 只写 `test_login_success`，覆盖率约 40%。
**正确做法：** 枚举每个 `if/else` 分支：成功路径 + 每种失败路径各一个函数。
提示词中 14+3+19=36 个函数名全部列出，让模型知道"每个分支必须有独立测试"。

---

### 意图十一：Web 前端不引用外部资源

> "纯 HTML + CSS + 原生 JS，不引用任何外部 CDN，无框架依赖"

外部 CDN（Bootstrap、Vue 等）在中国大陆网络下加载慢甚至失败，手机端体验极差。
所有样式和脚本内联在单个 HTML 文件中，离线也能正常运行，加载速度最快。

---

### 意图十二：前端 token 存储策略

> "token 存取：localStorage.getItem/setItem('access_token')"

JWT 适合无状态 API，存在 `localStorage` 让用户刷新页面后保持登录态（初始化时验证 token）。
提示词中明确说明初始化逻辑：有 token → 调用 `/users/me` 验证 → 成功则直接进 dashboard，
这个"自动恢复登录态"功能是 SPA 的基本体验，必须显式约定。

---

### 意图十三：CI/CD 串行依赖的安全性

> test → build → deploy 串行，各 Job 有 `needs` 依赖

**设计意图：**
- 测试失败 → 镜像不构建 → 服务器不更新（坏代码永远不进生产）
- PR 阶段只运行 Job 1（`if: push to main` 条件屏蔽 Job 2/3），避免 PR 因缺少 SSH Secrets 而报错
- 镜像同时打 `latest` 和 SHA 两个标签，方便回滚到任意历史版本

---

### 意图十四：localhost.run 方案的选择理由

提示词中收录了 localhost.run 而非 ngrok 的原因：
- ngrok 免费版**需要注册账号 + 配置 authtoken**，增加人工步骤，违反"无需人工干预"原则
- localhost.run 只需 **SSH 密钥**（`ssh-keygen` 自动生成），无需任何账号
- 提示词中包含了生成 SSH 密钥的命令，整个流程完全自动化

---

### 意图十五：`db.create_all()` 幂等初始化

> "幂等，每次启动安全建表"

SQLAlchemy 的 `create_all()` 只创建不存在的表，不破坏已有数据。
在 `run.py` 的 `app_context` 内调用，让容器每次启动都自动初始化数据库，
无需单独的 migration 步骤，适合 SQLite 这类轻量数据库场景。

---

### 意图十六：验证步骤作为"完成定义"

> 提示词末尾附上 5 条具体命令

给模型提供明确的"完成定义"：生成的代码必须能通过所有这些命令。
模型在生成过程中会隐式反向验证——
"`run:app` 能被 gunicorn 找到吗？`run.py` 里有 `app` 变量吗？"
"`pytest --cov-fail-under=100` 会通过吗？36 个分支都覆盖了吗？"
这种自我检查机制将输出质量提升到接近可直接运行的水平。
