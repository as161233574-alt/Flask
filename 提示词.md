## 一、数据模型

### User（表名 users）
字段：
  id            INTEGER  PRIMARY KEY AUTOINCREMENT
  username      VARCHAR(80)   UNIQUE NOT NULL
  email         VARCHAR(120)  UNIQUE NOT NULL
  password_hash VARCHAR(256)  NOT NULL
  role          VARCHAR(20)   NOT NULL DEFAULT 'user'   -- 只允许 'user' | 'admin'
  created_at    DATETIME      DEFAULT CURRENT_TIMESTAMP
  updated_at    DATETIME      DEFAULT CURRENT_TIMESTAMP

方法：
  set_password(plain)   使用 werkzeug generate_password_hash 存储
  check_password(plain) 使用 werkzeug check_password_hash 验证
  to_dict()             返回 {id, username, email, role, created_at（ISO格式）}，不含 password_hash

### TokenBlocklist（表名 token_blocklist）
字段：
  id         INTEGER  PRIMARY KEY AUTOINCREMENT
  jti        VARCHAR(36)  UNIQUE NOT NULL
  created_at DATETIME     DEFAULT CURRENT_TIMESTAMP

## 二、API 端点

所有响应格式统一：
  成功：{"success": true,  "data": {...}}
  失败：{"success": false, "error": "错误描述"}

HTTP 状态码约定：
  200 成功  201 创建成功  400 参数错误  401 未认证
  403 无权限  404 未找到  409 冲突（重复）

### 认证模块 Blueprint("auth")，前缀 /auth

POST /auth/register
  无需认证
  body: {username, email, password}
  校验：三个字段均不能为空；password 最少 6 位；username 和 email 均不能与已有记录重复
  成功：201 + user.to_dict()

POST /auth/login
  无需认证
  body: {email, password}
  校验：两个字段均不能为空；email + password 组合匹配
  成功：200 + {access_token, refresh_token}

POST /auth/logout
  需要 Bearer access token
  将当前 token 的 jti 写入 TokenBlocklist
  成功：200 + {message: "logged out"}

POST /auth/refresh
  需要 Bearer refresh token（@jwt_required(refresh=True)）
  成功：200 + {access_token}（新签发）

### 用户模块 Blueprint("user")，前缀 /users

GET /users/me
  需要 Bearer access token
  返回当前用户的 to_dict()

PUT /users/me
  需要 Bearer access token
  body: {username?, email?}（两个字段均为可选）
  有值时才更新，更新前检查唯一性冲突
  成功：200 + user.to_dict()

PUT /users/me/password
  需要 Bearer access token
  body: {current_password, new_password}
  校验：两个字段不能为空；current_password 必须正确；new_password 最少 6 位
  成功：200 + {message: "password updated"}

GET /users/
  需要 Bearer access token + role == 'admin'
  返回所有用户的 to_dict() 列表

PUT /users/<int:user_id>/role
  需要 Bearer access token + role == 'admin'
  body: {role}，只允许 'user' 或 'admin'
  目标用户不存在时返回 404
  成功：200 + user.to_dict()

### 健康检查

GET /health
  无需认证，返回 {"status": "ok"}，HTTP 200
  用于负载均衡器和容器健康探活

## 三、核心实现细节

### app/extensions.py
只做实例化，不绑定 app，避免循环导入：
  db = SQLAlchemy()
  jwt = JWTManager()

### app/config.py
三个配置类继承自基类 Config：

Config（基类）：
  SECRET_KEY       = os.environ.get("SECRET_KEY", "dev-secret-key")
  JWT_SECRET_KEY   = os.environ.get("JWT_SECRET_KEY", "dev-jwt-secret")
  SQLALCHEMY_TRACK_MODIFICATIONS = False

DevelopmentConfig：
  DEBUG = True
  SQLALCHEMY_DATABASE_URI = os.environ.get("DATABASE_URL", "sqlite:///users.db")

TestingConfig：
  TESTING = True
  SQLALCHEMY_DATABASE_URI = "sqlite:///:memory:"
  JWT_ACCESS_TOKEN_EXPIRES  = False
  JWT_REFRESH_TOKEN_EXPIRES = False

ProductionConfig：
  SQLALCHEMY_DATABASE_URI = os.environ.get("DATABASE_URL", "sqlite:////app/data/users.db")

config_map = {"development": DevelopmentConfig, "testing": TestingConfig, "production": ProductionConfig}

### app/__init__.py（App Factory）
create_app(env="development") 中按以下顺序执行：
1. 创建 Flask app，加载配置
2. 初始化 CORS：
   CORS(app, resources={r"/*": {"origins": "*"}}, supports_credentials=True, expose_headers=["Authorization"])
3. db.init_app(app)，jwt.init_app(app)
4. 注册 @jwt.token_in_blocklist_loader：
   查询 TokenBlocklist.query.filter_by(jti=jti).exists()，用 db.session.query(...).scalar() 返回布尔值
5. 注册 auth_bp（/auth）和 user_bp（/users）
6. 在 app 上直接注册 GET /health 路由，返回 {"status": "ok"}

### app/models/user.py（注意导入路径）
from ..extensions import db   # 注意是两个点，从 models 包向上到 app 包再找 extensions

### app/utils/decorators.py（admin_required）
使用 @wraps(fn) + @jwt_required() 组合：
- 通过 get_jwt_identity() 查询数据库获取用户
- user.role != 'admin' 时返回 403 JSON {"success": false, "error": "admin access required"}
- 通过则调用原函数

### run.py
  import os
  from app import create_app
  from app.extensions import db
  env = os.environ.get("FLASK_ENV", "development")
  app = create_app(env)
  with app.app_context():
      db.create_all()
  if __name__ == "__main__":
      app.run()

  if: github.event_name == 'push' && github.ref == 'refs/heads/main'

## 七、deploy.sh（服务器首次初始化脚本）

bash 脚本，开头 set -euo pipefail，按顺序执行：
1. 检查 docker / docker compose 是否可用
2. mkdir -p $DEPLOY_DIR（默认 /opt/flask-user-system）
3. 如果环境变量 REPO_URL 不为空，则 git clone 或 git pull
4. 检查 .env 是否存在，不存在则打印参考模板并 exit 1
5. export GHCR_OWNER（从环境变量或 .env 文件中读取）
6. docker compose -f docker-compose.prod.yml pull
7. docker compose -f docker-compose.prod.yml up -d --remove-orphans
8. docker image prune -f
9. sleep 5 && curl -sf http://localhost:8000/health，成功打印"服务正常运行"，失败打印日志查看命令

## 提示词设计意图详解

### 原则一：在开头明确禁止残缺输出

> "不得截断，不得使用省略号或 # TODO 占位符"

**问题根源：** 模型在上下文接近限制时会自动压缩输出，写出
`# ... rest of implementation ...` 或 `pass  # TODO` 这类无法运行的代码。
这是 AI 代码生成最常见的失败模式。

**解决方案：** 在提示词开头第一句就明确禁止，模型会将此作为约束条件贯穿整个输出。
如果上下文确实不够，模型会分多轮完成，而不是静默截断。

---

### 原则二：文件路径标记——让输出可直接机器解析

> "每个文件第一行写注释：# FILE: 相对路径"

**问题根源：** 当模型输出 15+ 个文件时，文件边界模糊，人工复制容易出错，
自动化脚本也难以可靠地拆分。

**解决方案：** 固定格式的首行注释让脚本能用一行正则
（`re.split(r'# FILE: (.+)', output)`）准确还原文件结构。

---

### 原则三：完整文件列表——消除结构猜测

> 列出包含所有 `__init__.py` 的完整文件树

**实际教训：** 本项目在开发中遇到过 `from .extensions import db`（错误）
被写成 `from ..extensions import db`（正确）的相对导入问题，
根因是模型对包层级不确定时会猜测路径。

明确列出每一个文件（包括空的 `__init__.py`）消除了所有歧义：
模型知道 `models/user.py` 在 `models/` 包内，其父包是 `app/`，
因此相对导入必须用 `..extensions`。

---

### 原则四：固定版本号——防止 API 兼容性问题

**实际差异举例：**
- Flask 2.x 用 `@app.route`，Flask 3.x 推荐 `@bp.get/@bp.post`
- Flask-JWT-Extended 3.x 用 `@jwt_required`（无括号），4.x 改为 `@jwt_required()`（有括号）
- SQLAlchemy 1.x 用 `Model.query.get(id)`，2.x 改为 `db.session.get(Model, id)`

固定版本确保模型生成的代码与 `requirements.txt` 中安装的库行为完全一致。

---

### 原则五：字段级约束——保证模型与测试对齐

**问题根源：** 模型在没有约束时会自行决定字段名（`created_at` vs `createdAt`）
和 `to_dict()` 的键名，而测试代码直接用 `r.get_json()["data"]["username"]` 访问这些键。
字段名不一致会导致测试失败，但错误信息是 `KeyError` 而非语义错误，难以定位。

**解决方案：** 在数据模型部分逐字段描述，包括 `to_dict()` 的返回结构，
让实现和测试用同一份"合同"。

---

### 原则六：统一响应格式——消除测试中的随机变量

> `{"success": true, "data": {...}}` / `{"success": false, "error": "..."}`

模型有时会返回 `{"token": ...}`，有时返回 `{"data": {"token": ...}}`，
取决于它"见过"的训练样本。统一规范后，测试中的
`r.get_json()["data"]["access_token"]` 这行代码可以稳定工作。

---

### 原则七：测试 fixture 的 scope 策略

**`scope="session"` 的 app + `scope="function"` 的 client** 是经过验证的最佳组合：

- `session` 级的 app：只初始化一次 Flask 应用（加载配置、注册蓝图），避免重复开销
- `function` 级的 client：每个测试用例独立的 in-memory SQLite，`create_all/drop_all` 配对，
  测试 A 注册的用户不会影响测试 B

如果两者都用 `session`，并发测试或测试顺序变化时会出现数据残留问题。

---

### 原则八：按分支枚举测试用例——强制 100% 覆盖

**错误做法：** "为 login 写测试" → 只写 `test_login_success`，覆盖率约 40%。

**正确做法：** "为 login 的每个分支写测试"：
```
test_login_success         → 执行 if user and user.check_password() 分支
test_login_missing_fields  → 执行 if not email or not password 分支
test_login_wrong_password  → 执行 if not user.check_password() 分支
test_login_unknown_user    → 执行 if not user 分支
```

提示词中逐一列出所有负向路径，让模型知道需要为每个 `if/else` 写独立函数。

---

### 原则九：admin_token fixture 的特殊说明

> "在 app_context 内将 role 改为 admin"

**循环依赖问题：** `admin_token` 需要一个 admin 用户，但"创建 admin"的 API 端点本身
就需要 admin 权限（`PUT /users/<id>/role` 需要 admin）。
不能通过 API 创建第一个 admin，必须直接操作数据库。

提示词明确说明了这个绕过方式，否则模型可能会生成一个调用自身尚未存在的 API 的 fixture。

---

### 原则十：CI/CD 三阶段串行设计

```
test → build → deploy
```

**`needs` 关键字的作用：**
- `build needs test`：测试失败则镜像不构建，坏代码不会进入镜像仓库
- `deploy needs build`：镜像未推送则不 SSH，服务器不会拉到不存在的 tag

**`if: github.event_name == 'push'` 的必要性：**
PR 时只有 `test` job 运行（因为 `build` 和 `deploy` 带了这个条件）。
如果去掉此条件，每个 PR 的 `build` job 会因为没有 `SSH_*` Secrets 而失败，
但这不是真正的错误，会干扰对 PR 状态的判断。

---

### 原则十一：`/health` 端点的工程价值

**不只是"检查服务是否存活"：**

1. **Docker healthcheck**：让容器编排知道何时真正就绪，
   避免 `docker compose up -d` 返回后服务还在启动中导致的短暂 502

2. **CI/CD 部署验证**：`deploy.sh` 的最后一步 `curl /health` 提供了部署后的自动验证，
   不需要人工登录服务器检查日志

3. **测试覆盖**：`test_health` 和 `test_cors_header` 覆盖了 `app/__init__.py`
   中 CORS 初始化和 `/health` 路由这两行代码，确保覆盖率达到 100%

---

### 原则十二：验证步骤作为"完成定义"

> 在提示词末尾附上具体的验证命令

这给模型提供了一个明确的"完成定义"：生成的代码必须能通过这些命令。
模型在输出过程中会反向验证自己的代码——
"这段 gunicorn 命令能找到 `run:app` 吗？`run.py` 里确实定义了 `app` 变量吗？"
这种隐式的自我检查机制显著提升了输出质量。
