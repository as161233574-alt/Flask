## 提示词设计意图说明

### 1. 角色设定（"你是一名资深后端工程师"）
**意图**：激活模型的专业代码生成模式，减少解释性废话，直接输出可运行代码。
不设定角色时模型倾向于给出教程式回答；设定后输出更紧凑、更专业。

### 2. 明确禁止截断（"不得截断，不得使用占位符"）
**意图**：防止模型在长文件中写 `# ... rest of code ...` 或 `# TODO`。
这是 AI 代码生成最常见的失败模式，必须在提示词开头明确禁止。

### 3. 完整文件列表（含 `__init__.py`）
**意图**：消除模型的"猜测空间"。列出每一个文件（包括空的 `__init__.py`）
可以防止模型遗漏文件或自行决定文件结构，确保输出可以直接 `cp -r` 使用。

### 4. 技术栈固定版本号
**意图**：避免版本兼容问题。Flask 3.x 和 Flask 2.x API 有差异，
JWT-Extended 4.x 和 3.x 也有 breaking change。固定版本确保代码与依赖一致。

### 5. 数据模型字段级描述
**意图**：模型在没有明确约束时会自行添加字段或省略字段。
逐字段描述（包括方法签名）确保生成的 ORM 模型与测试 fixture 完全匹配。

### 6. 端点表格 + 认证要求
**意图**：将 API 设计作为"合同"传递给模型。
明确每个端点的 HTTP 方法、路径、认证类型，防止模型自行发明端点或混淆认证逻辑。

### 7. 统一响应格式规范
**意图**：测试代码依赖固定的 JSON 结构（`data["access_token"]` 等）。
如果响应格式不统一，测试会随机失败。提前规范格式让测试和实现保持一致。

### 8. 测试 fixture 逐一描述
**意图**：conftest.py 是测试的基础设施，fixture 的 scope 和行为直接影响测试隔离性。
明确 `scope=function` + `create_all/drop_all` 确保每个测试用例使用干净的数据库。

### 9. 测试用例按分支枚举
**意图**：100% 覆盖率要求每个 if/else 分支都被测试。
逐一列出"成功、缺少字段、密码错误、用户不存在"等分支，
让模型知道需要为每个分支写独立的 `test_` 函数，而不是只写 happy path。

### 10. 验证步骤作为"验收标准"
**意图**：给模型一个明确的"完成定义"。
模型在生成代码时会反向验证：生成的代码能否通过这些步骤？
这种自我验证机制显著提升输出质量。

### 11. 文件标记注释（`# FILE: 路径`）
**意图**：当模型在单次响应中输出 15+ 个文件时，需要明确的文件边界标记，
方便用户或自动化脚本解析并写入正确路径。
